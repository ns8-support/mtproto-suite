## 📢 Reporting a Vulnerability

We take security seriously. If you discover a vulnerability:

### How to Report

**Email:** ns8sup@gmail.com

**GitHub:** https://github.com/mtproto-suite/mtproto-suite/security/advisories/new (private disclosure)

**DO NOT** open a public GitHub issue for security vulnerabilities.

### What to Include

1. **Description** of the vulnerability
2. **Steps to reproduce** (proof-of-concept)
3. **Affected versions**
4. **Impact assessment** (what attacker can achieve)
5. **Your contact info** (for follow-up questions)

### Response Timeline

| Stage | Timeline |
|---|---|
| Acknowledgment | Within 48 hours |
| Initial assessment | Within 7 days |
| Fix released | Within 30 days (critical: 7 days) |
| Public disclosure | After fix is widely deployed |

### Bounty Program

We do not currently offer a bug bounty program. Credit will be given in release notes for verified reports.

## 🔍 SSH Credentials Security Audit

A comprehensive audit of SSH credentials handling (logins, passwords, private keys) has been completed. See [`SECURITY_AUDIT_SSH.md`](SECURITY_AUDIT_SSH.md) for the full report.

### Audit Summary

**✅ Protected from credential leaks:**
- Passwords are NOT logged in plaintext
- Private keys are NOT logged
- Credentials transmitted per-request via HTTPS, never persisted
- JWT protects all endpoints
- All SSH commands are hardcoded (no command injection vector)
- Input validation (IPv4, port, token) before SSH connection
- SSH connection always closed via try/finally

**✅ Additional protections added:**
- CORS whitelist (only panel frontend can make requests)
- Rate limiting on SSH endpoints (10 requests per 5 minutes)
- Error message sanitization (no stack traces, IPs, paths leaked)
- Security headers (HSTS, X-Frame-Options, X-Content-Type-Options)
- Logger auto-redaction for password, token, privateKey, secret
- Global error handler separating server logs from client responses

See [`SECURITY_AUDIT_SSH.md`](SECURITY_AUDIT_SSH.md) for the detailed report.

## 🏗 Security Architecture

### Authentication Layers

**Layer 1: Panel API (JWT)**
- Algorithm: HS256
- Expiry: 24 hours (configurable)
- Secret: 64-char hex (256-bit entropy)
- Storage: PostgreSQL `users.password_hash` (bcrypt cost 12)
- Transport: HTTPS only (TLS 1.2/1.3)

**Layer 2: Service Node API (Bearer Token)**
- Static token, 64-char hex (256-bit)
- Comparison: `crypto.timingSafeEqual` (constant-time)
- Transport: HTTPS recommended
- Auto-generated by `install.sh --mode=node`

**Layer 3: SSH (Panel → Node)**
- Methods: Password OR OpenSSH private key
- Optional passphrase
- Connection: New SSH session per request (no persistent connection)
- Credentials: Transmitted per-request via HTTPS, never persisted

**Layer 4: Database (Internal)**
- PostgreSQL on private Docker network
- No external access (panel-only)
- Parameterized queries (SQL injection impossible)

**Layer 5: Docker (Internal)**
- Containers run on `mtproto-net` bridge network
- nginx uses host network (for port 443 + client IP visibility)
- Volume isolation (data persisted on host)

### Cryptographic Algorithms

| Purpose | Algorithm | Parameters |
|---|---|---|
| Password hashing | bcrypt | cost 12 (≈250ms per hash) |
| JWT signing | HMAC-SHA256 | 256-bit secret |
| Token comparison | timingSafeEqual | (built-in) |
| TLS | TLS 1.2 / 1.3 | Modern cipher suite |
| SSL (Let's Encrypt) | RSA 2048 | ACME v2 |
| SSH | ed25519 / RSA 4096 | OpenSSH format |

### Default Security Settings

✅ **Enabled by default:**
- JWT_SECRET must be set (no insecure default)
- AUTH_TOKEN must be ≥16 chars (no empty token allowed)
- HTTPS recommended in install.sh
- `Strict-Transport-Security` header
- `X-Frame-Options: SAMEORIGIN`
- `X-Content-Type-Options: nosniff`
- bcrypt cost 12 for admin password
- `timingSafeEqual` for token comparison
- Username enumeration protection
- `.env` files chmod 600
- Docker container `restart: unless-stopped`

### 📋 Recent Security Audit

A comprehensive audit of SSH credentials handling has been completed. See [`SECURITY_AUDIT_SSH.md`](SECURITY_AUDIT_SSH.md) for the full report (10 findings, all addressed).

**Audit summary:**

| Issue | Severity | Status |
|---|---|---|
| Error messages leaking internal IPs/paths | High | ✅ Fixed (`sanitizeErrorMessage`) |
| Default CORS allowed any origin | High | ✅ Fixed (whitelist via env var) |
| No rate limiting on SSH endpoints | High | ✅ Fixed (10 req/5min limiter) |
| Missing security headers (HSTS, X-Frame-Options, etc.) | Medium | ✅ Fixed |
| Logger could leak credentials in meta | Medium | ✅ Fixed (`sanitizeMeta`) |
| SSH connection not always closed on error | Medium | ✅ Fixed (try/finally) |
| No timeout on SSH commands | Medium | ✅ Fixed (30s timeout) |

❌ **Disabled by default (require explicit opt-in):**
- TLS for panel (HTTP only — for dev/testing)
- Public IP exposure of panel (relies on firewall)
- Self-signed certs (warning to user)

## 🎯 Threat Model

### Adversaries

**Tier 1: Casual attackers / script kiddies**
- Capability: Automated scanning
- Motivation: Low
- Mitigation: Standard firewall, no known exploits

**Tier 2: Hacktivists**
- Capability: Targeted attacks on Telegram-related infrastructure
- Motivation: Medium
- Mitigation: TLS, strong auth, monitoring

**Tier 3: Nation-state actors (RKN, GFW)**
- Capability: Deep packet inspection, traffic analysis
- Motivation: Block MTProto traffic
- Mitigation: fake TLS via SNI, VLESS/Reality VPN

**Tier 4: Insider threats**
- Capability: Access to admin panel or SSH
- Motivation: Sabotage, espionage
- Mitigation: Audit logs (planned), least privilege

### Threats Addressed

| Threat | Vector | Mitigation |
|---|---|---|
| **Credential brute-force** | `/api/auth/login` | bcrypt (slow), JWT expiry 24h, no password lockout (recommend fail2ban) |
| **Token theft** | XSS, MITM | HTTPS only, localStorage (not cookie), SameSite=Strict |
| **Timing attack on token** | Side-channel | `timingSafeEqual` |
| **Username enumeration** | Response timing | Identical bcrypt for unknown users |
| **SQL injection** | API requests | Parameterized queries (pg library) |
| **CSRF** | Browser requests | Bearer-token (not cookie-based) |
| **MITM on API** | Network | TLS 1.2/1.3 enforced |
| **Container escape** | Compromised proxy | Docker isolation, non-root user inside containers (planned) |
| **Fake TLS fingerprinting** | DPI | `tls_emulation: true`, custom cipher suites |
| **Brute-force SSH** | Panel → Node | SSH credentials not persisted (transmitted per-request) |
| **MITM on SSH** | Network | SSH protocol (built-in encryption) |
| **Replay attack on actions** | API | JWT includes expiry, token rotation |

### Threats NOT Addressed (Out of Scope)

- **Physical server access** — if attacker has root, game over
- **Compromised Docker daemon** — bypass container isolation
- **Compromised OS kernel** — bypass all isolation
- **Side-channel attacks** (Spectre, Meltdown) — require kernel patches
- **Telegram-side bans** — handled by ME servers (telemt feature)
- **Legal/regulatory** — user responsibility to comply with local laws

### Risk Acceptance

| Risk | Level | Accepted? |
|---|---|---|
| HTTP-only panel (dev mode) | High | Only for local development |
| Self-signed TLS | Medium | For internal networks |
| Long-lived SSH credentials | Medium | Mitigated by short-lived SSH sessions |
| No audit log (v2.0) | Medium | Planned for v2.1 |
| No rate limiting | Medium | Recommended `express-rate-limit` |
| No intrusion detection | Medium | Recommended Fail2ban |

## ✅ Security Best Practices

### For Panel Operators

1. **Use HTTPS** — never expose panel over HTTP in production
2. **Use Let's Encrypt** — not self-signed
3. **Strong admin password** — minimum 16 chars, random
4. **Restrict network access** — firewall panel to trusted IPs
5. **Regular updates** — keep panel and nodes up-to-date
6. **Backup `.env` files** — encrypted backup of secrets
7. **Monitor logs** — alert on suspicious activity
8. **Use SSH keys** — disable password SSH on nodes
9. **Rotate tokens** — regenerate AUTH_TOKEN periodically
10. **Audit access** — review who has panel access

### For Node Operators

1. **Disable password SSH** — `PasswordAuthentication no` in `/etc/ssh/sshd_config`
2. **Use SSH keys with passphrase** — protects against key theft
3. **Enable UFW** — only open necessary ports
4. **Enable automatic security updates** — `unattended-upgrades`
5. **Monitor Docker** — `docker events` for container lifecycle
6. **Restrict Docker socket** — root-only access
7. **Use fail2ban** — protects against SSH brute-force
8. **Audit users** — `last`, `lastlog`, `/var/log/auth.log`
9. **Patch kernel** — `apt upgrade linux-image-*` regularly
10. **Backup data** — daily snapshots of `/opt/mtproto-suite`

### For End Users (Telegram clients)

1. **Use official Telegram app** — verified, no backdoors
2. **Verify proxy URL** — paste in Telegram, not browser
3. **Use VPN in parallel** — for double privacy
4. **Avoid sharing tg:// links** — each user should have unique proxy

## 🔧 Hardening Guide

### Panel Hardening

#### 1. HTTPS Only

```bash
# Reinstall with Let's Encrypt
bash install.sh --mode=panel --ssl-letsencrypt panel.example.com

# Or manually: redirect HTTP to HTTPS in nginx
# Edit panel-frontend/nginx.conf:
server {
    listen 80;
    server_name panel.example.com;
    return 301 https://$server_name$request_uri;
}
```

#### 2. Firewall Rules

```bash
# UFW (Ubuntu/Debian)
ufw default deny incoming
ufw default allow outgoing
ufw allow 22/tcp     # SSH (only from your IP)
ufw allow 443/tcp    # HTTPS

# Restrict by IP (replace YOUR_IP with actual IP)
ufw allow from YOUR_IP to any port 22

# iptables (CentOS/RHEL)
iptables -A INPUT -p tcp --dport 443 -j ACCEPT
iptables -A INPUT -p tcp --dport 22 -s YOUR_IP -j ACCEPT
iptables -A INPUT -j DROP
```

#### 3. Fail2ban

```bash
# Install
apt-get install fail2ban

# Enable SSH jail (default)
systemctl enable fail2ban
systemctl start fail2ban

# Verify
fail2ban-client status sshd
```

#### 4. Strong Admin Password

```bash
# Generate
openssl rand -base64 32

# Update via panel UI: Settings → Change Password
# Or via SQL (if locked out):
docker exec -it <db-container> psql -U mtproto -d mtproto_panel -c "
UPDATE users SET password_hash = crypt('NEW_STRONG_PASSWORD', gen_salt('bf', 12))
WHERE username = 'admin';
"
```

#### 5. Rotate Secrets

```bash
# JWT_SECRET (requires panel restart)
NEW_SECRET=$(openssl rand -hex 32)
sed -i "s/JWT_SECRET=.*/JWT_SECRET=$NEW_SECRET/" /opt/mtproto-suite/panel-backend/.env
cd /opt/mtproto-suite && docker compose restart backend

# All users must re-login (existing tokens invalidated)

# DB_PASSWORD (more invasive)
NEW_PASS=$(openssl rand -hex 16)
sed -i "s/DB_PASSWORD=.*/DB_PASSWORD=$NEW_PASS/" /opt/mtproto-suite/panel-backend/.env
# Also update in docker-compose.yml or .env file
cd /opt/mtproto-suite && docker compose down && docker compose up -d

# AUTH_TOKEN (per node)
NEW_TOKEN=$(openssl rand -hex 32)
# Edit on node: /opt/mtproto-suite/service-node/.env
# Update in panel: Nodes → Edit → New token
```

### Node Hardening

#### 1. SSH Key Auth Only

```bash
# On node
sed -i 's/^#PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
sed -i 's/^PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
systemctl restart sshd

# Test (should still work with your key)
ssh root@<node-ip> "echo 'OK'"
```

#### 2. Disable Root Login (Use Sudo User)

```bash
# Create sudo user
adduser mtproto
usermod -aG sudo mtproto
mkdir -p /home/mtproto/.ssh
cp ~/.ssh/authorized_keys /home/mtproto/.ssh/
chown -R mtproto:mtproto /home/mtproto/.ssh

# Disable root SSH
sed -i 's/^#PermitRootLogin yes/PermitRootLogin no/' /etc/ssh/sshd_config
sed -i 's/^PermitRootLogin yes/PermitRootLogin no/' /etc/ssh/sshd_config
systemctl restart sshd

# In panel, change SSH username from "root" to "mtproto"
```

#### 3. UFW on Node

```bash
ufw default deny incoming
ufw default allow outgoing

# SSH from panel IP only
ufw allow from <panel-ip> to any port 22

# Proxy port (public)
ufw allow 443/tcp

# Node API (from panel only)
ufw allow from <panel-ip> to any port 8443

# Enable
ufw enable
```

#### 4. Docker Socket Permissions

```bash
# Default: socket is rw for root and docker group
# Only root and docker group should have access

# Add users to docker group (NOT recommended)
usermod -aG docker mtproto  # Now mtproto can run docker

# Better: use sudo for docker access
echo 'mtproto ALL=(ALL) NOPASSWD: /usr/bin/docker' >> /etc/sudoers.d/mtproto
```

#### 5. Automatic Security Updates

```bash
# Ubuntu/Debian
apt-get install unattended-upgrades
dpkg-reconfigure -plow unattended-upgrades

# CentOS/RHEL
yum install yum-cron
systemctl enable yum-cron
```

### Application Hardening

#### 1. Rate Limiting (Recommended)

```typescript
// panel-backend/src/index.ts
import rateLimit from 'express-rate-limit';

const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutes
  max: 10,                     // 10 attempts
  message: { error: 'Too many login attempts' },
});

app.use('/api/auth/login', authLimiter);
```

#### 2. Security Headers

Already configured in `panel-frontend/nginx-ssl.conf`:
- `Strict-Transport-Security: max-age=31536000; includeSubDomains`
- `X-Frame-Options: SAMEORIGIN`
- `X-Content-Type-Options: nosniff`

#### 3. CORS Hardening

```typescript
// panel-backend/src/index.ts
// Currently: app.use(cors()) — allows ALL origins
// Recommended: restrict to known origins
app.use(cors({
  origin: ['https://panel.example.com'],
  credentials: true,
}));
```

## 📋 Compliance

### GDPR

MTProto Suite does **not collect personal data** beyond:
- Admin username (single user)
- Node IP addresses (operational)
- IP addresses of proxy users (for geo-location, optional)

All data is stored locally. No telemetry. No third-party analytics.

### Data Retention

| Data | Retention | Justification |
|---|---|---|
| Proxy configs | Until deleted | Operational |
| Stats history | 7 days | Troubleshooting |
| IP history | Until deleted | Operational, optional |
| Admin password | Until changed | Authentication |
| SSH credentials | Per-request (not stored) | Security |
| JWT tokens | 24 hours (in browser localStorage) | Authentication |
| Logs | 30 MB (rotated) | Debugging |

### User Rights (GDPR)

If you operate MTProto Suite in EU:
- Right to access: Provide admin user list (just yourself)
- Right to deletion: Delete users from `users` table
- Right to portability: Export data via `pg_dump`

### Russian 152-FZ (Personal Data Law)

If operating in Russia:
- IP addresses are personal data
- Storage must be on Russian servers (if collecting)
- We don't recommend using MTProto Suite for collecting personal data

## 🔍 Security Audit Checklist

Run through this checklist before production deployment:

### Panel

- [ ] HTTPS with valid certificate (not self-signed)
- [ ] JWT_SECRET is 64-char hex (256-bit)
- [ ] Admin password is 16+ chars, random
- [ ] DB_PASSWORD is 32-char hex
- [ ] `.env` files have 600 permissions
- [ ] Firewall limits panel access to trusted IPs
- [ ] fail2ban installed and running
- [ ] No SSH password authentication (keys only)
- [ ] SSH root login disabled (use sudo user)
- [ ] Audit logs reviewed for suspicious activity

### Nodes

- [ ] AUTH_TOKEN is 64-char hex (256-bit)
- [ ] UFW limits SSH to panel IP
- [ ] UFW limits API (8443) to panel IP
- [ ] Proxy port (443) accessible from internet
- [ ] SSH keys only (no password auth)
- [ ] Docker socket accessible only by docker group
- [ ] Automatic security updates enabled
- [ ] Daily backups of data directory

### Code

- [ ] No `console.log` of secrets
- [ ] No `.env` files in git
- [ ] No hardcoded credentials
- [ ] All SQL queries parameterized
- [ ] All user input validated (IP, port, token format)
- [ ] CORS restricted to known origins
- [ ] Rate limiting on auth endpoints
- [ ] Dependencies up-to-date (no known CVEs)

### Operational

- [ ] Backups stored offsite (S3, encrypted)
- [ ] Disaster recovery plan documented
- [ ] Monitoring + alerting configured
- [ ] Incident response plan documented
- [ ] Security contact email published
- [ ] All admin actions logged (planned for v2.1)
