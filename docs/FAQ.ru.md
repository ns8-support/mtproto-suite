## 🌐 Общие

### Что такое MTProto Suite?

MTProto Suite — это единый Docker-проект для управления MTProto Telegram прокси в масштабе. Он объединяет:
- **Web панель** (UI) для централизованного управления
- **Сервис-ноды** (runtime) на каждом прокси-сервере
- Опциональное **VPN-туннелирование** (VLESS/Reality) для обхода блокировок

### Чем он отличается от оригинальных репозиториев `danielVNru`?

| Аспект | Оригинал | MTProto Suite |
|---|---|---|
| Репозитории | 2 (panel + node) | 1 единый |
| Безопасность | JWT дефолт `'change-me-in-production'` | Обязателен, без дефолта |
| Мониторинг | Базовый (только Docker stats) | Полный CPU/RAM/Disk + история |
| Удалённая установка | ❌ | ✅ Через SSH |
| SSL | Вручную | Автоматический Let's Encrypt / Cloudflare |
| NetBird | ❌ | ✅ Mesh VPN |
| Docker compose | Один на сервис | Единый |
| Тесты | Нет | (в процессе) |

### Можно ли использовать это коммерчески?

Да! MIT лицензия разрешает коммерческое использование. Просто сохраните copyright notice.

### Это связано с Telegram?

Нет. Это независимый open-source проект. "MTProto" и "Telegram" — торговые марки Telegram Messenger LLP.

## 📦 Установка

### Какие дистрибутивы Linux поддерживаются?

- Ubuntu 20.04, 22.04, 24.04
- Debian 11, 12
- CentOS / RHEL 8, 9
- AlmaLinux / Rocky Linux 8, 9

Только x86_64 и aarch64 (64-битный ARM). 32-битные системы не поддерживаются.

### Можно ли установить на macOS или Windows?

Технически да для **панели** (Docker Desktop), но не рекомендуется для production. macOS/Windows не могут использоваться как **сервис-ноды** (прокси серверы).

### Нужно ли устанавливать Node.js?

**Нет!** Все компоненты работают в Docker контейнерах. Единственное требование — Docker Engine 20.10+ и Docker Compose plugin.

### Сколько места на диске нужно?

- **Панель**: 2 GB минимум
- **Каждая нода**: 5 GB минимум + 100 MB на контейнер прокси
- **SSL сертификаты**: < 1 MB

### Можно ли установить панель и ноду на одной машине?

Да! Используйте `--mode=both` для тестирования или домашнего использования. Для production используйте отдельные серверы.

### Какой минимум RAM?

- **Панель**: 1 GB (рекомендуется 2 GB)
- **Каждая нода**: 512 MB (1 GB на 50 прокси)

## ⚙️ Конфигурация

### Где хранятся файлы конфигурации?

| Файл | Расположение |
|---|---|
| Панель .env | `/opt/mtproto-suite/panel-backend/.env` |
| Нода .env | `/opt/mtproto-suite/service-node/.env` |
| Конфиги прокси (нода) | `/opt/mtproto-suite/service-node/data/store.json` |
| История статистики | `/opt/mtproto-suite/service-node/data/stats-history.json` |
| История IP | `/opt/mtproto-suite/service-node/data/ip-history.json` |
| SSL сертификаты | `/opt/mtproto-suite/ssl/wildcard/` |

### Можно ли использовать внешний PostgreSQL?

Да. Установите `DB_HOST` на внешний хост в `panel-backend/.env`. Установщик ожидает PostgreSQL 16+, но более старые версии могут работать.

### Как изменить порт панели?

1. Отредактируйте `/opt/mtproto-suite/panel-backend/.env`: `PORT=<новый-порт>`
2. Перезапустите: `cd /opt/mtproto-suite && docker compose restart frontend`
3. Обновите правила firewall

### Как добавить кастомные fake TLS домены?

Через UI панели: **"Domains"** → ввести список через запятую → Сохранить.
Или через API:
```bash
curl -X PUT http://node:8443/api/domains \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"domains":["google.com","microsoft.com"]}'
```

### Как ограничить подключения на прокси?

При создании/редактировании прокси установите `maxConnections`. nginx применит директиву `limit_conn`.

## 🌐 Прокси

### В чём разница между ME и DC?

**ME (Middle End) серверы** — Telegram прокси, используемые для начального подключения. Обходят прямую блокировку IP.

**DC (Data Center)** — Фактические серверы Telegram. Заблокированы в некоторых странах.

**telemt** автоматически использует ME для обфускации, при необходимости переключается на DC.

### Что такое fake TLS?

Техника, при которой трафик прокси выглядит как HTTPS к легитимному домену (например, `google.com`). Telegram клиенты подключаются к порту 443 на вашем сервере, но соединение на самом деле MTProto, замаскированное под TLS для обхода DPI.

### Зачем мне нужен VLESS/Reality VPN?

В России, Китае, Иране и других странах DC Telegram заблокированы. ME серверы частично работают, но паттерны трафика можно обнаружить.

**VLESS/Reality** маршрутизирует весь трафик через VPN сервер в незаблокированной стране (Нидерланды, Германия и т.д.). Трафик выглядит как легитимный TLS к популярному сайту (например, `google.com`), что делает блокировку невозможной.

### Сколько прокси на ноду?

Зависит от ресурсов хоста:
- 1 vCPU / 1 GB RAM: ~10 прокси
- 2 vCPU / 2 GB RAM: ~50 прокси
- 4 vCPU / 4 GB RAM: ~100 прокси
- 8 vCPU / 8 GB RAM: ~300 прокси

### Можно ли запускать одно и то же прокси на нескольких нодах?

Да! Экспортируйте с одной ноды, импортируйте на другую. Каждая нода получает отдельный контейнер, но URL прокси (tg://) тот же.

### Как поделиться прокси с пользователями?

1. В панели найдите прокси
2. Нажмите **"Скопировать tg:// ссылку"**
3. Поделитесь ссылкой с пользователями (Telegram, веб-сайт и т.д.)
4. Пользователи открывают её в Telegram → Настройки → Прокси → Вставить

## 🔒 Безопасность

### Мои данные зашифрованы?

- **В передаче**: HTTPS (TLS 1.2/1.3)
- **В покое**: Пароль админа хешируется через bcrypt. Другие секреты (JWT_SECRET, AUTH_TOKEN) хранятся в открытом виде в файлах `.env` (только права файловой системы).
- **База данных**: PostgreSQL в приватной Docker сети

Для production используйте шифрование дисков (LUKS, BitLocker) и менеджер секретов (Vault).

### Что если кто-то получит мой AUTH_TOKEN?

Они могут получить доступ к API ноды. Митигация:
1. Ротировать токен: `openssl rand -hex 32`
2. Обновить в UI панели
3. Обновить в `.env` ноды
4. Перезапустить: `docker compose restart`

### Должен ли я выставлять панель в интернет?

Только если необходимо. Лучшая практика:
- Запускайте панель в приватной сети (например, 10.0.0.0/16)
- Доступ через VPN (WireGuard, NetBird)
- Или используйте bastion host с MFA

### Что если я потеряю пароль админа?

Сброс через PostgreSQL:
```bash
docker exec -it <db-container> psql -U mtproto -d mtproto_panel -c "
UPDATE users SET password_hash = crypt('новый-пароль', gen_salt('bf', 12))
WHERE username = 'admin';
"
```

### Как часто нужно ротировать секреты?

- **JWT_SECRET**: Каждые 90 дней (инвалидирует все сессии)
- **AUTH_TOKEN**: Каждые 180 дней
- **DB_PASSWORD**: Каждые 365 дней
- **Пароль админа**: Каждые 90 дней
- **SSH ключи**: Каждые 2 года

### Хранятся ли где-то SSH credentials?

**Нет.** SSH credentials передаются per-request из frontend в panel backend через HTTPS, используются для установления единственного SSH-соединения с целевой нодой, и затем собираются сборщиком мусора (GC) после завершения запроса.

- ❌ Не сохраняются в PostgreSQL
- ❌ Не сохраняются в localStorage (frontend)
- ❌ Не сохраняются в cookies
- ✅ Только в React component state (теряются при reload страницы)
- ✅ Logger автоматически редактирует любые credentials в логах

См. [`SECURITY_AUDIT_SSH.md`](SECURITY_AUDIT_SSH.md) для детального аудита.

### Что произойдёт если панель обнаружит нарушение CORS?

С версии 2.0.0, panel API использует CORS whitelist через `PANEL_FRONTEND_URL`. Если запрос приходит с origin не из whitelist:

1. Запрос отклоняется с `CORS: origin not allowed`
2. Заблокированный origin логируется через `logger.warn('cors', ...)`
3. В консоли разработчика браузера появится CORS ошибка

Это защищает от cross-origin атак, где вредоносные сайты пытаются вызывать API панели используя JWT пользователя.

### Какие есть rate limits?

С версии 2.0.0, rate limiting применяется через `express-rate-limit`:

| Endpoint | Лимит |
|---|---|
| Все endpoints (глобально) | 200 req/min на IP |
| `/api/auth/login` | 10 попыток за 15 мин на IP |
| SSH endpoints (`/api/remote-install/*`, `/api/nodes/:id/*`) | 10 req за 5 мин на IP |

Превышение лимита возвращает HTTP 429 с `{"error": "Too many requests"}`.

### Могу ли я увидеть internal error details для отладки?

По дизайну **нет** — сообщения об ошибках, возвращаемые клиенту, санитизируются для предотвращения утечки информации. Полные детали ошибок (с stack traces, internal IP, путями файлов) доступны только в server logs:

```bash
# Просмотр логов panel backend
docker logs -f mtproto-panel-backend

# Или по конкретной категории
docker logs mtproto-panel-backend 2>&1 | grep '"category":"panel.proxy"'
```

## ⚡ Производительность

### Насколько быстр MTProto Suite?

Пропускная способность зависит от:
- CPU ноды (telemt привязан к CPU)
- Пропускная способность сети между нодой и DC Telegram
- Используется ли VPN (добавляет задержку)

Типично: **1-10 MB/s на прокси**, **50-200 MB/s на ноду**.

### Почему моё прокси медленное?

Распространённые причины:
1. **CPU узкое место**: Слишком много прокси на хосте
2. **Сетевое узкое место**: ISP ограничивает пропускную способность
3. **Overhead VPN**: VLESS добавляет 50-200мс задержки
4. **Блокировка DPI**: Используйте fake TLS или VLESS для маскировки

### Как масштабировать более 100 нод?

- **Вертикально**: Увеличить ресурсы хоста панели (CPU, RAM, тюнинг PostgreSQL)
- **Горизонтально**: Добавить больше инстансов панели за load balancer (сложно)
- **Шардинг**: Одна панель на регион/дата-центр

Для 1000+ нод рассмотрите:
- PostgreSQL read replicas
- Redis кеш для частых запросов
- Prometheus для агрегации метрик

### Влияет ли NetBird на производительность?

NetBird добавляет ~1-5мс задержки для mesh-роутинга. Незаметно для прокси трафика.

### Должен ли я использовать SSD или HDD?

**SSD настоятельно рекомендуется** для нод. nginx access логи и запись IP истории I/O-интенсивны.

## 🐛 Troubleshooting

### Установщик завершается с ошибкой "Docker не найден"

Установите Docker вручную:
```bash
curl -fsSL https://get.docker.com | sh
systemctl enable --now docker
```

### Моё прокси перестаёт работать через несколько часов

Проверьте счётчик перезапусков контейнера:
```bash
docker ps -a --filter "name=mtproto-proxy"
# Ищите "Restarting" или высокое количество перезапусков в колонке Status

Распространённые причины:
- OOM (нехватка памяти) — увеличьте RAM хоста
- Таймаут сети — проверьте лимиты ISP
- Бан Telegram DC — ротируйте IP, используйте VPN
```

### tg:// ссылка работает в Telegram, но прокси медленное

Используйте ME серверы (по умолчанию) вместо прямого DC. Установите `useMiddleProxy: true` и `me2dcFallback: true`.

### В логах вижу "ECONNREFUSED"

Проверьте:
1. Контейнер запущен: `docker ps | grep mtproto-proxy`
2. Порт открыт: `netstat -tlnp | grep 443`
3. Firewall разрешает: `ufw status`

### Как включить debug логирование?

Установите `LOG_LEVEL=debug` в соответствующем файле `.env`, перезапустите контейнер.

## 📚 Глоссарий

**MTProto** — Нативный протокол Telegram для связи клиент-сервер.

**ME (Middle End)** — Промежуточные прокси-серверы Telegram для обфускации соединения.

**DC (Data Center)** — Фактические серверы Telegram для маршрутизации сообщений.

**SNI (Server Name Indication)** — Расширение TLS, указывающее к какому домену клиент хочет подключиться (используется для fake TLS).

**fake TLS** — MTProto трафик, замаскированный под HTTPS для обхода DPI.

**VLESS** — Современный VPN протокол на базе XTLS, устойчивый к обнаружению.

**Reality** — Расширение TLS от XTLS, имитирующее легитимные сайты (например, `google.com`).

**ACME** — Automatic Certificate Management Environment (протокол, используемый Let's Encrypt).

**DNS-01 challenge** — Метод валидации домена, требующий создания TXT записей (используется для wildcard сертификатов).

**ACME client** — Программа, реализующая протокол ACME (мы используем `acme-client` для Node.js).

**WireGuard** — Современный VPN протокол (используется NetBird под капотом).

**Mesh VPN** — Децентрализованный VPN, где пиры подключаются друг к другу (без центрального сервера после настройки).

**NetBird** — Open-source mesh VPN на базе WireGuard.

**SOCKS5** — Протокол прокси (xray в нашей настройке слушает на SOCKS5 порт 10808).

**proxychains** — Linux утилита, которая заставляет TCP соединения проходить через SOCKS прокси (telemt использует это для доступа к Telegram через xray).

**fake TLS** — Маскировка MTProto под легитимный HTTPS трафик.

**tg://proxy** — Протокол Telegram для обмена конфигурациями прокси.

**JWT (JSON Web Token)** — Формат токена для stateless аутентификации (алгоритм HS256 с общим секретом).

**Bearer token** — Схема аутентификации, где токен передаётся в заголовке `Authorization: Bearer <token>`.

**bcrypt** — Функция хеширования паролей (намеренно медленная, с настраиваемым cost).

**timingSafeEqual** — Функция сравнения с константным временем (предотвращает timing атаки).

**SSH (Secure Shell)** — Зашифрованный протокол для удалённого доступа к серверу.

**DHCP** — Dynamic Host Configuration Protocol (здесь не используется).

**Docker** — Среда исполнения контейнеров (мы используем 20.10+).

**docker-compose** — Инструмент для многоконтейнерных Docker приложений (мы используем плагин версии 2.0+).
