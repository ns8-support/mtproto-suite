## 🔑 Начальный вход

После установки вы увидите credentials в выводе:

```
✓ Панель установлена и запущена

URL:        https://panel.example.com
Логин:      admin
Пароль:     ********
```

1. Откройте URL в браузере
2. Введите логин и пароль
3. Нажмите **"Войти"**
4. JWT токен сохраняется в `localStorage` на 24 часа

> 🔒 **Смените пароль по умолчанию немедленно**, если используете автоматически сгенерированный в общей среде.

## 🖥️ Управление нодами

### Страница нод

Показывает все зарегистрированные сервис-ноды с:
- Имя, IP, порт
- Статус (онлайн/офлайн через периодическую проверку здоровья)
- Количество прокси на ноде
- Время последнего обновления

### Добавление ноды

**Способ 1: Вручную (существующая service-node)**

1. Нажмите **"+ Добавить ноду"**
2. Заполните:
   - **Имя**: Понятное имя (например, "Франкфурт #1")
   - **IP**: Публичный IP или домен
   - **Порт**: 8443 (по умолчанию)
   - **Токен**: Из вывода `install.sh --mode=node`
   - **Домен** (опционально): Для `tg://proxy?...` ссылок
3. Нажмите **"Проверить соединение"** для проверки
4. Нажмите **"Добавить ноду"**

**Способ 2: Удалённая SSH установка (рекомендуется для новых серверов)**

1. Нажмите **"🛠 Установить удалённо"**
2. Введите SSH credentials:
   - Хост (IP или домен)
   - Порт (по умолчанию 22)
   - Имя пользователя (по умолчанию `root`)
   - Пароль ИЛИ private key
3. Нажмите **"🔌 Проверить соединение"** — проверяет SSH и показывает информацию об ОС
4. Настройте:
   - Порт API ноды (по умолчанию 8443)
   - Порт прокси (по умолчанию 443)
   - NAT_IP (для VPN режима)
5. Нажмите **"🚀 Установить"** — запускает полный pipeline установки
6. После успеха нажмите **"+ Добавить как ноду"**

### Действия с нодой

Нажмите на ноду чтобы увидеть детали. Доступные действия:

| Действие | Описание |
|---|---|
| 📊 **Мониторинг** | Метрики CPU/RAM/Disk/Docker в реальном времени |
| 🔄 **Обновить** | Триггерит `update.sh` на ноде |
| 🗑️ **Удалить** | Удаляет из БД панели (не удаляет с сервера) |
| 🔐 **Переподключить** | Переподключение с новым токеном |

## 🌐 Создание прокси

### Через UI

1. Откройте страницу деталей ноды
2. Нажмите **"+ Создать прокси"**
3. **Основные настройки:**
   - **Имя**: Понятное имя
   - **Домен**: Авто-выбирается из пула (или кастомный)
   - **Макс. подключений** (опционально): Лимит одновременных устройств
4. **Расширенные настройки** (нажмите чтобы раскрыть):
   - VPN подписка (VLESS URL)
   - Mask host (редирект не-MTProto трафика)
   - Кастомные опции telemt (30+ параметров)
5. Нажмите **"Создать"**

Прокси теперь активен. Чтобы получить ссылку для отправки:

1. Найдите прокси в списке
2. Нажмите **"📋 Скопировать tg:// ссылку"**
3. Поделитесь с Telegram клиентами

### Состояния прокси

| Статус | Значение |
|---|---|
| `running` | Активен, принимает соединения |
| `paused` | Приостановлен через UI (можно возобновить) |
| `stopped` | Остановлен вручную или ошибка контейнера |
| `error` | Контейнер не найден (вероятно крашнулся) |

### Изменение прокси

1. Откройте страницу деталей прокси
2. Отредактируйте нужные поля
3. Нажмите **"Сохранить"**

Некоторые изменения требуют перезапуск контейнера:
- Домен
- VPN подписка
- Tag
- Кастомные опции telemt

Перезапуск происходит автоматически при необходимости.

### Пауза vs Удаление

**Пауза** — временно отключает прокси (контейнер приостановлен, состояние сохранено).
**Удаление** — полностью удаляет прокси (контейнер остановлен и удалён, статистика удалена).

### Массовый экспорт / импорт

Экспорт всех прокси с одной ноды для миграции:

1. Откройте детали ноды
2. Нажмите **"📦 Экспорт"**
3. Сохраните `proxies-export.json`

Импорт на другую ноду:

1. Откройте детали целевой ноды
2. Нажмите **"📥 Импорт"**
3. Загрузите `proxies-export.json`
4. Просмотрите импортированные прокси

## 📊 Мониторинг нод

### Метрики в реальном времени

При нажатии на ноду вы видите:

**Информация о системе:**
- Hostname, ОС, версия ядра, архитектура
- IP адреса
- Аптайм
- Текущее время сервера

**Метрики (авто-обновление):**
- Использование CPU (с load average и количеством ядер)
- Использование RAM (total/used/free, swap)
- Использование диска (для каждой точки монтирования)
- Статистика Docker контейнеров (CPU%, RAM, сеть)

**Интервал обновления:** 5с / 10с / 30с / 1м / выкл (настраивается в UI)

### Графики истории

Нажмите кнопки **"1ч" / "6ч" / "24ч" / "7д"** для просмотра исторических метрик:
- Использование CPU по времени
- Использование RAM по времени
- Использование диска по времени
- Количество работающих контейнеров

История хранится в PostgreSQL (последние 1000 точек на ноду).

### Статистика контейнеров

Детальный просмотр каждого Docker контейнера на ноде:
- Имя, образ, статус
- CPU%, память (использовано/лимит)
- Сеть I/O (RX/TX)

### Оповещения и предупреждения

Страница мониторинга использует цветовую кодировку:
- 🟢 Зелёный (<50%): Здоровый
- 🟡 Жёлтый (50-75%): Наблюдать
- 🟠 Оранжевый (75-90%): Предупреждение
- 🔴 Красный (>90%): Критично

## ⚡ Удалённые действия

### Перезапуск Service Node

1. Откройте детали ноды
2. Нажмите **"🔄 Перезапустить сервис"**
3. Подтвердите
4. Панель выполняет `docker compose restart service-node` через SSH
5. Ждёт до 60 секунд для health check
6. Сообщает об успехе/ошибке с логами

### Перезагрузка сервера

> ⚠️ **ДЕСТРУКТИВНАЯ ОПЕРАЦИЯ** — все запущенные процессы будут остановлены.

1. Откройте детали ноды
2. Нажмите **"🔌 Перезагрузить сервер"**
3. **Диалог подтверждения:** Введите имя ноды точно
4. Панель выполняет `sudo reboot` через SSH
5. Соединение с нодой пропадёт на 1-3 минуты
6. После перезагрузки нода должна автоматически переподключиться

**Требования:**
- `sudo` с NOPASSWD в `/etc/sudoers`
- Или root SSH доступ (root имеет парольный sudo по умолчанию)

### Заметки по безопасности SSH-операций

Когда вы нажимаете любую из кнопок выше, панель:

1. Открывает новое SSH-соединение к целевой ноде (per-request)
2. Выполняет захардкоженную команду (нет command injection)
3. Закрывает соединение сразу после (try/finally)
4. Логирует только hostname/username ноды (никогда пароль или privateKey)
5. Возвращает sanitized ошибку если операция провалилась

**Ваши SSH credentials:**
- ✅ Передаются только через HTTPS
- ✅ Никогда не сохраняются в БД, browser storage или cookies
- ✅ Собираются сборщиком мусора после каждой операции
- ❌ НЕ кешируются — нужно вводить заново при reload страницы

**Rate limiting:** 10 SSH-запросов за 5 минут на IP. Если вы превысите лимит, увидите "Too many SSH requests" — подождите 5 минут или свяжитесь с админом.

См. [`SECURITY_AUDIT_SSH.md`](SECURITY_AUDIT_SSH.md) для детального аудита.

### Установка NetBird

1. Откройте детали ноды
2. Прокрутите до секции **"NetBird"**
3. Введите:
   - **Setup Key** из NetBird dashboard
   - **Management URL** (опционально, для self-hosted)
   - **Hostname** в mesh-сети (опционально)
4. Нажмите **"🚀 Установить & Подключиться"**

После успеха нода получает mesh IP (например, `100.64.0.5`).

### Удаление NetBird

1. Откройте детали ноды → секция NetBird
2. Нажмите **"🗑️ Удалить"**
3. Подтвердите

## 🔒 Wildcard SSL через Cloudflare

### Требования

1. **Домен добавлен в Cloudflare** (бесплатного тарифа достаточно)
2. **API Token** с правами:
   - `Zone:DNS:Edit` — создание/удаление TXT записей
   - `Zone:Zone:Read` — поиск zone_id

Создайте токен на: https://dash.cloudflare.com/profile/api-tokens
Используйте шаблон: **"Edit zone DNS"**

### Получение wildcard сертификата

1. Откройте панель → страница **SSL Сертификаты**
2. Нажмите **"+ Получить wildcard"**
3. Введите Cloudflare API Token
4. Нажмите **"🔑 Проверить токен"** — проверяет и показывает доступные домены
5. Выберите домен из списка (или введите вручную)
6. Введите email для Let's Encrypt
7. Опционально включите **Staging** (для тестирования)
8. Нажмите **"🔒 Получить wildcard сертификат"**

**Процесс:**
1. ACME аккаунт создан на Let's Encrypt
2. CSR сгенерирован для `*.example.com`
3. TXT запись `_acme-challenge.example.com` создана через Cloudflare
4. DNS-01 challenge пройден
5. Сертификат и приватный ключ сохранены в `/opt/mtproto-suite/ssl/wildcard/`
6. TXT запись удалена

**Результат:**
```
✓ Сертификат успешно получен!

Действителен до: 2026-10-04 (90 дней)
Сертификат: /opt/mtproto-suite/ssl/wildcard/example_com.cert.pem
Приватный ключ: /opt/mtproto-suite/ssl/wildcard/example_com.key.pem
```

### Использование wildcard сертификата

**Для nginx:**
```nginx
ssl_certificate /opt/mtproto-suite/ssl/wildcard/example_com.cert.pem;
ssl_certificate_key /opt/mtproto-suite/ssl/wildcard/example_com.key.pem;
```

**Для сервис-нод:**
```bash
# Скопировать на ноду
scp /opt/mtproto-suite/ssl/wildcard/example_com.* root@node1:/opt/mtproto-suite/ssl/

# Использовать для SNI маскировки
```

### Автообновление

Let's Encrypt сертификаты действительны 90 дней. Настройте cron для автообновления:

```bash
# Редактировать crontab
crontab -e

# Добавить эту строку (обновляется еженедельно, только если истекает)
0 3 * * 1 cd /opt/mtproto-suite/ssl/wildcard && \
  bash /opt/mtproto-suite/install.sh --ssl-renew example.com
```

## 🛡️ VLESS/Reality VPN

### Что это делает

VLESS/Reality — это устойчивый к цензуре VPN протокол. В MTProto Suite:

**Без VPN:**
```
Клиент → Service-node:443 → Telegram DC
```

**С VPN (hybrid режим):**
```
Клиент → Service-node:443 → xray (SOCKS5) → VLESS сервер → Telegram DC
                              ↑
                              ME трафик через tun0 → EU IP (обход РКН)
```

### Настройка

**1. Получите VLESS подписку:**

От вашего VPN провайдера (например, 3x-ui, x-ui, marzban), получите:
- **URL подписки**: `https://provider.com/sub/abc123`
- **Сырая vless:// ссылка**: `vless://uuid@host:port?...`

**2. Настройте прокси:**

1. Откройте панель → Ноды → Создать/Редактировать прокси
2. Прокрутите до секции **"VPN"**
3. Вставьте URL подписки или vless:// ссылку
4. (Опционально) Установите **NAT_IP** = публичный IP вашего VPN exit сервера
5. Нажмите **"Сохранить"**

Панель:
1. Скачает и распарсит VLESS конфиг
2. Создаст xray контейнер с этим конфигом
3. Настроит telemt использовать его как upstream SOCKS5 прокси
4. Перезапустит telemt контейнер

### Поддерживаемые транспорты

| Транспорт | Случай использования |
|---|---|
| `tcp` | Прямой TCP (наименее устойчив к DPI) |
| `ws` | WebSocket (хорошая маскировка) |
| `grpc` | gRPC (современный, труднее детектить) |
| `xhttp` | XHTTP (новейший, лучшая маскировка) |

### Режимы

**Hybrid режим** (NAT_IP + VLESS):
- ME трафик: прямой через host routing → tunnel → EU IP
- DC трафик: через xray SOCKS5 → VLESS → Telegram DC

**Legacy режим** (только VLESS):
- ME трафик: прямой
- DC трафик: через xray SOCKS5

**Simple режим** (только NAT_IP):
- Весь трафик: прямой через tun0 → EU IP
- Используйте когда есть VPN туннель, но нет VLESS

## 📦 Массовые операции

### Экспорт всех прокси с ноды

```bash
# Через API
curl -fsS http://node:8443/api/export \
  -H "Authorization: Bearer <token>" \
  > proxies-export.json

# Через UI: детали ноды → кнопка Export
```

### Импорт на другую ноду

```bash
# Через UI: целевая нода → Import → загрузить файл
```

Поведение при импорте:
- Каждый прокси из файла создаётся на целевой ноде
- Ошибки собираются и показываются в конце
- Возможен частичный успех

### Стратегия бэкапа

```bash
# Ежедневный бэкап всех нод
for node in $(cat nodes.txt); do
  ssh root@$node "cd /opt/mtproto-suite && \
    tar czf /backup/node-\$(date +%Y%m%d).tar.gz \
      service-node/data service-node/.env"
done
```

## 📅 Ежедневные операции

### Проверка здоровья

```bash
# Все ноды сразу
for node in $(cat nodes.txt); do
  echo "=== $node ==="
  curl -fsS http://$node:8443/api/health \
    -H "Authorization: Bearer $(cat token-$node)" \
    | jq .
done
```

### Обновление всех прокси

```bash
# Обновить конкретный прокси
curl -X PUT http://panel/api/nodes/1/proxies/abc123 \
  -H "Authorization: Bearer $(cat jwt)" \
  -H "Content-Type: application/json" \
  -d '{"maxConnections": 100}'
```

### Пауза проблемных прокси

```bash
# Приостановить все прокси с высоким количеством соединений
curl http://panel/api/proxies/all \
  -H "Authorization: Bearer $(cat jwt)" | \
  jq '.[] | .proxies[] | select(.connectedIps | length > 50) | .id' | \
  while read id; do
    curl -X POST http://panel/api/nodes/1/proxies/$id/pause \
      -H "Authorization: Bearer $(cat jwt)"
  done
```

### Мониторинг через API

```bash
# Метрики в реальном времени
watch -n 5 'curl -fsS http://node:8443/api/proxies \
  -H "Authorization: Bearer $(cat token)" \
  | jq ".[] | {name, status, connectedIps: (.connectedIps | length)}"'
```

## 🔍 Частые задачи

### Найти прокси по домену

```bash
curl -fsS http://node:8443/api/proxies \
  -H "Authorization: Bearer $(cat token)" | \
  jq '.[] | select(.domain == "google.com")'
```

### Получить tg:// ссылку

```bash
curl -fsS "http://node:8443/api/proxies/abc123/link?server_ip=proxy.example.com" \
  -H "Authorization: Bearer $(cat token)"
# Возвращает: {"link": "tg://proxy?server=proxy.example.com&port=443&secret=..."}
```

### Экспорт конфигурации прокси

```bash
# Конфиг одного прокси
curl -fsS http://node:8443/api/proxies/abc123 \
  -H "Authorization: Bearer $(cat token)" | jq .
```

### Ротация IP Blacklist

```bash
# Получить текущий blacklist
curl http://node:8443/api/blacklist \
  -H "Authorization: Bearer $(cat token)"

# Обновить blacklist
curl -X PUT http://node:8443/api/blacklist \
  -H "Authorization: Bearer $(cat token)" \
  -H "Content-Type: application/json" \
  -d '{"ips": ["1.2.3.4", "5.6.7.8"]}'
```

### Изменение кастомных доменов

```bash
curl -X PUT http://node:8443/api/domains \
  -H "Authorization: Bearer $(cat token)" \
  -H "Content-Type: application/json" \
  -d '{"domains": ["google.com", "microsoft.com", "apple.com"]}'
```
